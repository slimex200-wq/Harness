## CLAUDE.md Rules

### NEVER
- NEVER use `setState` in a `StatelessWidget`. Use `StatefulWidget` or Riverpod providers. **Why:** 컴파일 에러 + 아키텍처 위반.
- NEVER import Firebase packages directly from UI layer (`lib/ui/`). Always go through repository layer (`lib/data/`). **Why:** 레이어드 아키텍처 위반. Firebase 교체 시 UI 전체 수정 필요.
- NEVER hardcode Firebase project IDs, API keys, or google-services.json values in source code. Use `--dart-define` or `.env` files. **Why:** 환경별 배포 불가 + 보안 사고.
- NEVER run `firebase deploy` without explicit confirmation. **Why:** 프로덕션 배포는 되돌리기 어려움.
- NEVER commit `pubspec.lock` changes without reviewing dependency diffs. **Why:** 의도치 않은 breaking change 유입.
- NEVER use `Opacity` widget for fade effects. Use `AnimatedOpacity` or `FadeTransition`. **Why:** Opacity는 매 프레임 saveLayer 호출 -> 성능 저하.
- NEVER create giant `build()` methods (50+ lines). Extract sub-widgets. **Why:** 리빌드 범위 확대 + 가독성 파괴.
- NEVER skip `const` constructor when all fields are final and compile-time constant. **Why:** const 누락은 불필요한 리빌드 유발.
- NEVER use `print()` for logging. Use `debugPrint()` or `logger` package. **Why:** print는 릴리즈 빌드에서도 출력되고, 긴 문자열 잘림.
- NEVER inline `pubspec.lock` content in prompts or context. **Why:** 수천 줄짜리 파일이 컨텍스트 예산 낭비.

### ALWAYS
- ALWAYS search pub.dev before implementing custom solutions. `dart pub search <keyword>` or web search. **Why:** 40k+ 패키지 생태계. 바퀴 재발명 방지.
- ALWAYS use `const` constructors wherever possible. **Why:** Flutter 리빌드 최적화의 기본.
- ALWAYS run `dart analyze` with zero warnings before commit. **Why:** 린트 경고 누적은 기술부채.
- ALWAYS run `dart format` on every `.dart` file before commit. **Why:** 일관된 코드 스타일 유지.
- ALWAYS specify `platform_flag` (android|ios|web|all) in every task observation. **Why:** 플랫폼별 동작 차이 추적 필수.
- ALWAYS use `ListView.builder` for dynamic lists (not `ListView` with children). **Why:** 메모리 효율 + 16ms 프레임 예산 준수.
- ALWAYS wrap Firebase calls in try-catch with structured error types. **Why:** Firebase 네트워크 에러는 예측 불가.
- ALWAYS follow MVVM + Repository pattern: `lib/ui/` -> `lib/data/` -> `lib/di/`. **Why:** 테스트 용이성 + 관심사 분리.
- ALWAYS use Riverpod for app-wide state management. `ChangeNotifier` only for isolated local state. **Why:** compile-safe, auto-dispose, no BuildContext 의존.
- ALWAYS indicate `hot_reload_safe: true|false` when reporting code changes. **Why:** StatefulWidget 상태 변경 등은 hot restart 필요.

### Workflow
```
Research (pub.dev, Firebase docs) -> Plan -> TDD (RED -> GREEN -> IMPROVE, 80%+ coverage) -> Code Review -> Commit
```

1. **Research**: pub.dev 패키지 검색, Firebase 문서 확인, 기존 솔루션 조사
2. **Plan**: 요구사항 정리, 아키텍처 결정, 테스트 전략 수립
3. **TDD**:
   - RED: 실패하는 테스트 작성 (widget test / unit test / integration_test)
   - GREEN: 테스트 통과하는 최소 코드 작성
   - IMPROVE: 리팩토링 + `flutter test --coverage` 80%+ 확인
4. **Code Review**: flutter-review 스킬로 Dart-specific 검사
5. **Commit**: `dart format` + `dart analyze` 통과 후 conventional commit

### Agents (Auto-Use)
- Complex feature -> **planner** -> Code written -> **flutter-review**
- Bug fix / new feature -> **flutter-tdd**
- Architecture decision -> **flutter-arch**
- Build fail -> **build-error-resolver** (flutter clean -> rebuild, max 2 cycles)
- Before commit -> **security-reviewer** (Firebase rules, API keys, env secrets)
- 3+ files changed -> independent **flutter-verify** agent (implementer != verifier)
- Firebase operations -> **firebase-guard** (deploy/delete confirmation)

### Custom Skills (Generate)

#### 1. flutter-tdd
- **Purpose**: Flutter-specific TDD extending generic tdd skill
- **Content**: Widget test (WidgetTester/pumpWidget), unit test (repositories/viewmodels with mockito), integration_test (E2E flows). Golden test support. `flutter test --coverage` 80%+ gate. Platform-conditional test selection (android|ios|web|all).
- **Path**: `skills/flutter-tdd/SKILL.md`

#### 2. flutter-review
- **Purpose**: Flutter/Dart code review with framework-specific checks
- **Content**: const constructor usage, widget decomposition (no giant build methods), setState in StatelessWidget check, Riverpod provider scoping, Firebase security rule validation, platform-conditional logic review, performance anti-pattern detection (Opacity, missing keys in lists).
- **Path**: `skills/flutter-review/SKILL.md`

#### 3. flutter-verify
- **Purpose**: Pre-completion verification loop for Flutter projects
- **Content**: `dart analyze` (zero warnings) -> `dart format --set-exit-if-changed` -> `flutter test --coverage` (80%+ gate) -> `flutter build` (debug + platform check) -> pubspec.yaml version consistency -> l10n completeness check. Structured pass/fail report.
- **Path**: `skills/flutter-verify/SKILL.md`

#### 4. firebase-guard
- **Purpose**: Safety guard for Firebase operations requiring confirmation
- **Content**: PreToolUse hook intercepting `firebase deploy`, `firebase functions:delete`, `firebase firestore:delete`, `firebase auth:import/export`. Shows target project/environment, requires explicit confirmation, logs the operation. Rollback hints per operation type.
- **Path**: `skills/firebase-guard/SKILL.md`

#### 5. flutter-arch
- **Purpose**: Enforce layered MVVM architecture conventions
- **Content**: Directory structure validation (lib/ui/, lib/data/, lib/di/), import boundary checks (UI cannot import Firebase directly), Riverpod provider placement verification, naming convention enforcement. Violation reports with fix suggestions.
- **Path**: `skills/flutter-arch/SKILL.md`

### flutter-review-agent
```yaml
name: flutter-review-agent
description: Flutter/Dart 코드 리뷰 에이전트. 구현자와 분리된 독립 검증.
trigger: 3+ .dart files changed
prompt: |
  Flutter/Dart 코드 리뷰 수행. 다음 항목 검사:
  1. const constructor 사용 여부
  2. widget 분해 (build() 50줄 이하)
  3. StatelessWidget에 setState 사용 여부
  4. Riverpod provider scoping 적정성
  5. Firebase import가 UI layer에서 직접 사용되는지
  6. performance anti-patterns (Opacity, missing keys, unnecessary rebuilds)
  7. dart analyze 경고 0건
  대상 파일: {{changed_files}}
  결과: pass/fail + 위반 목록 + fix suggestions
```

## Commands

### /flutter-check
```
dart analyze && dart format --set-exit-if-changed . && flutter test --coverage
```
Quick validation: lint + format + test in one command.

### /flutter-clean-rebuild
```
flutter clean && flutter pub get && dart run build_runner build --delete-conflicting-outputs && flutter build {{platform}} --debug
```
Full clean rebuild with codegen. Use after dependency changes.

### /firebase-emulator
```
firebase emulators:start --project=demo-{{project_name}}
```
Start Firebase emulators for local development.

### /coverage-report
```
flutter test --coverage && genhtml coverage/lcov.info -o coverage/html && echo "Coverage report: coverage/html/index.html"
```
Generate and open HTML coverage report.

### /arch-check
```
Run flutter-arch-agent on lib/ directory to verify MVVM + Repository pattern compliance.
```

## Hooks Scripts

### scripts/dart-format-hook.sh
```bash
#!/bin/bash
# PostToolUse: Auto dart format on .dart file save
set -euo pipefail

FILEPATH="$1"
if [[ "$FILEPATH" == *.dart ]]; then
  dart format "$FILEPATH" 2>/dev/null || true
fi
```


```bash
set -euo pipefail
```

### scripts/firebase-guard-hook.sh
```bash
#!/bin/bash
# PreToolUse: Block firebase deploy without explicit confirmation
set -euo pipefail

echo "BLOCKED: firebase deploy requires explicit user confirmation."
echo "Target project: $(firebase use 2>/dev/null || echo 'unknown')"
echo "To proceed, user must confirm in chat."
exit 1
```

### scripts/release-guard-hook.sh
```bash
#!/bin/bash
# PreToolUse: Verify signing config before release build
set -euo pipefail

if [[ ! -f "android/key.properties" ]]; then
  echo "BLOCKED: android/key.properties not found. Cannot build release."
  exit 1
fi

echo "Signing config found. Release build allowed after user confirmation."
exit 1  # Still require explicit confirmation
```

### scripts/pre-commit-hook.sh
```bash
#!/bin/bash
# PreCommit: dart format + dart analyze + gitleaks
set -euo pipefail

echo "=== Pre-commit checks ==="

# 1. dart format check
echo "[1/3] dart format --set-exit-if-changed ."
dart format --set-exit-if-changed . || {
  echo "FAIL: dart format found unformatted files. Run: dart format ."
  exit 1
}

# 2. dart analyze
echo "[2/3] dart analyze"
dart analyze --fatal-infos || {
  echo "FAIL: dart analyze found issues."
  exit 1
}

# 3. gitleaks
echo "[3/3] gitleaks detect"
if command -v gitleaks &>/dev/null; then
  gitleaks detect --source . --no-git -v 2>&1 | tail -5 || {
    echo "FAIL: gitleaks detected secrets."
    exit 1
  }
else
  echo "WARN: gitleaks not installed. Skipping secret scan."
fi

echo "=== All pre-commit checks passed ==="
```
