# PRD: DailyLoop - 습관 추적 앱

## 개요
- **한 줄 설명**: 매일 반복하는 습관을 체크하고 연속 달성(스트릭)을 시각화하는 미니멀 습관 추적 앱
- **대상 사용자**: 자기 관리에 관심 있는 20-40대, 새해 목표나 루틴을 만들고 싶은 일반 사용자
- **핵심 가치**: "오늘 할 일을 체크하는 3초" -- 최소한의 마찰로 습관 달성을 기록하고, 연속 기록이 동기를 유지시킨다

## 왜 습관 추적 앱인가
1. **매일 사용**: 습관 앱은 본질적으로 매일 열어야 하는 앱이다 (DAU/MAU 비율이 높은 카테고리)
2. **Firebase 적합성**: 사용자별 문서 구조가 Firestore의 컬렉션/문서 모델에 자연스럽게 매핑된다
3. **달성 가능한 범위**: 핵심 기능(습관 CRUD + 일별 체크 + 스트릭 계산)이 명확하고 한정적이다
4. **크로스 플랫폼 가치**: 모바일(Android)에서 체크하고 데스크톱(Windows)에서 통계를 확인하는 시나리오가 자연스럽다

## 기술 스택

| 계층 | 기술 | 용도 |
|------|------|------|
| Frontend | Flutter 3.x + Dart | Android / Windows 크로스 플랫폼 UI |
| 인증 | Firebase Auth | 이메일/Google 소셜 로그인 |
| 데이터베이스 | Cloud Firestore | 습관 정의, 일별 체크 기록 |
| 알림 | Flutter Local Notifications | 리마인더 (FCM 불필요, 로컬 처리) |
| 상태관리 | Riverpod 2.x | 반응형 상태 관리 |
| 라우팅 | GoRouter | 선언적 네비게이션 |

## 데이터 모델 (Firestore)

```
users/{uid}/
  habits/{habitId}
    name: string
    icon: string (material icon name)
    color: string (hex)
    frequency: "daily" | "weekdays" | "custom"
    customDays: number[] (0-6, 월-일)
    reminderTime: timestamp | null
    createdAt: timestamp
    archivedAt: timestamp | null
    order: number

  checks/{YYYY-MM-DD}
    completedHabits: string[] (habitId 배열)
    updatedAt: timestamp
```

### Must Have (MVP)

| ID | 기능 | 사용자 스토리 | 수용 기준 |
|----|------|--------------|----------|
| M1 | 이메일/Google 로그인 | 사용자로서 계정을 만들어 데이터를 보존하고 싶다 | Firebase Auth로 이메일+비밀번호, Google Sign-In 지원. 로그아웃/재로그인 시 데이터 유지 |
| M2 | 습관 생성 | 사용자로서 추적할 습관을 추가하고 싶다 | 이름(필수), 아이콘, 색상, 반복 주기(매일/평일/커스텀) 설정. 최대 20개 제한 |
| M3 | 오늘의 체크리스트 | 사용자로서 오늘 해야 할 습관을 보고 완료 체크하고 싶다 | 홈 화면에 오늘 날짜의 활성 습관 목록 표시. 탭으로 완료/미완료 토글. 체크 즉시 Firestore 반영 |
| M4 | 스트릭 표시 | 사용자로서 며칠 연속 달성했는지 보고 싶다 | 각 습관 옆에 현재 연속 달성 일수 표시. 하루라도 빠지면 0으로 리셋 |
| M5 | 주간 캘린더 뷰 | 사용자로서 이번 주 달성 현황을 한눈에 보고 싶다 | 홈 화면 상단에 7일(월-일) 도트 그리드. 완료일은 색상 채움, 미완료는 빈 원 |
| M6 | 습관 편집/삭제 | 사용자로서 습관 이름이나 설정을 변경하고 싶다 | 습관 수정 시 이름/아이콘/색상/주기 변경 가능. 삭제 시 확인 다이얼로그 후 soft delete(archive) |
| M7 | 오프라인 지원 | 사용자로서 인터넷 없이도 체크하고 싶다 | Firestore 오프라인 퍼시스턴스 활성화. 오프라인 체크 후 온라인 복귀 시 자동 싱크 |

## 비기능 요구사항

| 항목 | 기준 |
|------|------|
| 앱 시작 시간 | 콜드 스타트 2초 이내 (캐시된 Firestore 데이터 기준) |
| 체크 응답 시간 | 탭 후 UI 반영 100ms 이내 (낙관적 업데이트, Firestore 싱크는 비동기) |
| 오프라인 동작 | Firestore 오프라인 퍼시스턴스로 네트워크 없이 전체 기능 사용 가능 |
| 데이터 보존 | 사용자 삭제 요청 전까지 모든 체크 기록 유지 (soft delete) |
| 플랫폼 지원 | Android 8.0+ (API 26), Windows 10+ |
| 접근성 | Material 3 기본 접근성 준수 (충분한 터치 타겟 48dp, 색상 대비 4.5:1) |
| 보안 | Firestore Rules로 사용자 본인 데이터만 읽기/쓰기 제한 |
| APK 크기 | 30MB 이하 (릴리즈 빌드 기준) |

## Firestore Security Rules

```javascript
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    match /users/{userId}/{document=**} {
      allow read, write: if request.auth != null && request.auth.uid == userId;
    }
  }
}
```

## 제외 범위 (Won't)

| 항목 | 이유 |
|------|------|
| iOS 빌드 | Windows 개발 환경에서 iOS 빌드 불가 (Xcode 필요) |
| 소셜/친구 기능 | 범위 초과. 개인 추적에 집중 |
| 복잡한 백엔드 (Cloud Functions) | Firebase 클라이언트 SDK만으로 구현. 서버리스 함수 불필요 |
| 유료 결제/프리미엄 | MVP 단계에서 수익화 불필요 |
| 웹 버전 | Flutter Web 퍼포먼스 이슈. Android + Windows에 집중 |
| AI 기반 추천 | 범위 초과. 단순 CRUD + 통계에 집중 |
| 반복 주기 "n일마다" | 복잡도 대비 사용 빈도 낮음. daily/weekdays/custom(요일 지정)으로 충분 |
| 푸시 알림 (FCM) | 로컬 알림으로 충분. FCM은 서버 인프라 필요 |
